# SQL Solutions Explained

## Описание

В этом файле собраны пояснения к SQL-блоку тестового задания Effective Mobile.

Задачи проверяют базовые навыки аналитического SQL: оконные функции, понимание `FULL JOIN`, агрегацию, `GROUP BY`, `HAVING`, фильтрацию по датам и связь бизнес-условия с запросом.

## Задача 1. Абитуриенты

### Условие

Есть таблица `examination`:

```sql
id      -- id абитуриента
scores  -- количество набранных баллов
```

Нужно создать колонку с позицией абитуриента в общем рейтинге.

### Решение

```sql
SELECT
    id,
    scores,
    RANK() OVER (ORDER BY scores DESC) AS position
FROM examination;
```

### Логика решения

В задаче нужно построить рейтинг абитуриентов по количеству набранных баллов. Чем больше баллов, тем выше позиция в рейтинге. Поэтому сортировка выполняется по `scores DESC`.

Для расчёта позиции используется оконная функция `RANK()`.

Оконная функция позволяет добавить рассчитанное значение к каждой строке, не сворачивая таблицу через `GROUP BY`. В результате сохраняются все абитуриенты, а к каждой строке добавляется позиция в рейтинге.

### Почему `RANK()`, а не `ROW_NUMBER()`

`ROW_NUMBER()` просто нумерует строки по порядку. Если два абитуриента набрали одинаковое количество баллов, они всё равно получат разные номера.

Для рейтинга это не всегда корректно. Если два человека набрали одинаковое количество баллов, они должны занимать одинаковую позицию.

### Почему `RANK()`, а не `DENSE_RANK()`

`DENSE_RANK()` тоже присваивает одинаковый ранг одинаковым значениям, но не оставляет пропусков после совпадений.

`RANK()` оставляет пропуск в позициях. Например, если два абитуриента делят первое место, следующий находится на третьей позиции. Для общего конкурсного рейтинга это часто более логично.

### Что сказать на собеседовании

Я использовал `RANK()`, потому что в рейтинге одинаковые баллы должны давать одинаковую позицию. При этом после совпадающих значений сохраняется пропуск в рангах, что отражает реальное место абитуриента в общем списке.

## Задача 2. FULL JOIN

### Условие

Есть две таблицы:

- первая содержит 30 строк;
- вторая содержит 20 строк.

Нужно определить возможный диапазон количества строк после `FULL JOIN`, если ключи могут полностью совпадать или быть полностью уникальными.

### Краткий ответ

При уникальных ключах:

```text
минимально 30 строк, максимально 50 строк
```

Если возможны дубликаты ключей и many-to-many соединение:

```text
минимально 30 строк, максимально 600 строк
```

### Логика решения

`FULL JOIN` возвращает все строки из обеих таблиц:

- совпавшие строки объединяются в одну результирующую строку;
- строки из первой таблицы без совпадений всё равно попадают в результат;
- строки из второй таблицы без совпадений тоже попадают в результат.

Если совпадений нет, результат будет состоять из всех строк первой и всех строк второй таблицы:

```text
30 + 20 = 50 строк
```

Если все 20 строк второй таблицы находят соответствия среди строк первой таблицы, то результат не может быть меньше размера большей таблицы:

```text
минимум = 30 строк
```

### Оговорка про дубликаты

Если ключи не уникальны, результат может увеличиться из-за many-to-many соединения.

Например, если все 30 строк первой таблицы имеют один ключ и все 20 строк второй таблицы имеют тот же ключ, каждая строка первой таблицы соединится с каждой строкой второй:

```text
30 * 20 = 600 строк
```

### Что сказать на собеседовании

При уникальных ключах диапазон результата `FULL JOIN` будет от 30 до 50 строк. Минимум равен размеру большей таблицы, максимум равен сумме строк двух таблиц. Если в ключах есть дубликаты, может возникнуть many-to-many эффект, и максимум может вырасти до 600 строк.

## Задача 3. Покупки

### Условие

Есть таблица счетов:

```sql
account(
    id integer,
    client_id integer,
    open_dt date,
    close_dt date
)
```

Есть таблица транзакций:

```sql
transaction(
    id integer,
    account_id integer,
    transaction_date date,
    amount numeric(10,2),
    type varchar(3)
)
```

Нужно вывести ID клиентов, которые за последний месяц по всем своим счетам совершили покупок меньше чем на 5000 рублей.

Ограничение: без подзапросов и оконных функций.

### Решение

```sql
SELECT
    a.client_id
FROM account a
JOIN transaction t
    ON a.id = t.account_id
WHERE t.type = 'BUY'
  AND t.transaction_date >= CURRENT_DATE - INTERVAL '1 month'
GROUP BY a.client_id
HAVING SUM(t.amount) < 5000
ORDER BY SUM(t.amount);
```

### Логика решения

В таблице `account` хранится связь между клиентом и его счетами. Один клиент может иметь несколько счетов.

В таблице `transaction` хранятся операции по счетам. Чтобы понять, какие транзакции относятся к какому клиенту, нужно соединить таблицы:

```sql
a.id = t.account_id
```

После соединения каждая транзакция становится связанной не только со счётом, но и с клиентом.

Так как в условии сказано "по всем своим счетам", группировать нужно по клиенту:

```sql
GROUP BY a.client_id
```

Тогда `SUM(t.amount)` считает общую сумму покупок клиента по всем его счетам.

### Почему используется `HAVING`

`WHERE` фильтрует отдельные строки до группировки.

А условие:

```sql
SUM(t.amount) < 5000
```

относится уже к агрегированной сумме по клиенту. Поэтому оно должно находиться в `HAVING`.

### Оговорка про клиентов без покупок

Если нужно считать, что клиент без покупок за последний месяц тоже подходит под условие "меньше 5000", запрос следует изменить:

```sql
SELECT
    a.client_id
FROM account a
LEFT JOIN transaction t
    ON a.id = t.account_id
   AND t.type = 'BUY'
   AND t.transaction_date >= CURRENT_DATE - INTERVAL '1 month'
GROUP BY a.client_id
HAVING COALESCE(SUM(t.amount), 0) < 5000
ORDER BY COALESCE(SUM(t.amount), 0);
```

В этом случае фильтры по транзакциям должны находиться в `ON`, а не в `WHERE`, чтобы `LEFT JOIN` не превратился в обычный `INNER JOIN`.

### Что сказать на собеседовании

Я соединяю `account` и `transaction`, чтобы связать транзакции со счетами и клиентами. Так как у клиента может быть несколько счетов, группировка выполняется по `client_id`, а не по `account_id`. После фильтрации покупок за последний месяц считаю сумму `amount` по каждому клиенту и через `HAVING` оставляю только тех, у кого сумма покупок меньше 5000 рублей.
